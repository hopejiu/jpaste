Strong P1

C1: main.go 职责过多（God Object）

LOC: ~240

files: main.go

Problem — main.go 当前承担了所有「装配」职责：创建 4 个 Service、5 个适配器（clipReader / clipWriter / eventEmitter / pasteRunner / toggleWindow）、注册全局热键、配置系统托盘、设置窗口事件钩子、清理过期数据。理解启动流程需要在同一个 240 行的文件中上下跳转。

Solution — 将装配逻辑按「启动阶段」抽取为独立函数，每个阶段读起来像一句英语。例如：

func main() {
    app, svcs := bootstrap()
    win := createWindow(app)
    setupSystemTray(app, win)
    registerGlobalHotkey(win, svcs.settings)
    runCleanup(svcs.history, svcs.settings)
    app.Run()
}

Benefits — Locality: 每个阶段的代码集中在一个函数，修改系统托盘不会碰到热键注册。 Leverage: 顶层 main() 成为可读的「脚本」，一眼看到启动做了几步。

Before
main() // ~240 lines
 ├─ init settings
 ├─ init DB
 ├─ 5 adapters
 ├─ 4 services
 ├─ app creation
 ├─ wire adapters
 ├─ window creation
 ├─ paste wiring
 ├─ system tray (menu, events)
 ├─ hotkey registration
 ├─ window hooks (close, blur)
 ├─ cleanup
 ├─ autostart
 └─ app.Run()
After
main() // ~20 lines
 ├─ bootstrap()       // app + services
 ├─ createWindow()    // + paste wiring
 ├─ setupSystemTray()
 ├─ registerHotkey()
 └─ runCleanup()
Strong P1

C2: 四个相同模式的 *application.App 适配器

4 × ~8 LOC each

files: main.go (clipReader, clipWriter, eventEmitter, pasteRunner)

Problem — clipReader、clipWriter、eventEmitter、pasteRunner 是四个几乎相同的结构体，都持有 *application.App（或 fn func()），在应用创建后再注入。这是同一个「懒装配」模式复制了四次。

Solution — 统一为一个 AppHandle 结构体，在 app 创建后用一次 Wire(app) 完成所有注入：

type AppHandle struct {
    app *application.App
}
func (h *AppHandle) ReadClipboard() (string, bool) { return h.app.Clipboard.Text() }
func (h *AppHandle) WriteClipboard(s string) bool  { return h.app.Clipboard.SetText(s) }
func (h *AppHandle) Emit(name string, data any)    { h.app.Event.Emit(name, data) }
func (h *AppHandle) Wire(app *application.App)     { h.app = app }

Benefits — Locality: 所有 app 依赖集中在一个结构体。 Leverage: 新增 app 依赖时只需加一个方法，不需要再创建新的 struct。 Deletion test: 删除 3 个 struct 定义，复杂度只有一个结构体。

C3: ClipboardService 职责过载

LOC: ~120

files: internal/clipboard/service.go, internal/db/db.go

Problem — ClipboardService 同时做了四件事：轮询剪贴板、计算 SHA-256、写 SQLite、发事件。轮询逻辑和持久化逻辑耦合在一起，测试剪贴板检测必须带着数据库。

Solution — 拆为两层：
① Watcher：纯轮询 + 哈希比较，产出 <-chan Entry
② Persister（并入 HistoryService）：消耗 channel，写入 DB + 发出事件
Watcher 可以用 mock clipboard reader 做单元测试，不依赖 DB。

Benefits — Locality: 轮询逻辑和存储逻辑可以独立修改。 Leverage: Watcher 的 channel 成为自然的 seam，未来可以接多个 consumer。

Before
ClipboardService (does everything)
 ├─ pollLoop() { ticker }
 ├─ checkClipboard()
 │   ├─ read clipboard
 │   ├─ sha256()
 │   ├─ upsert DB
 │   └─ emit events
 └─ ServiceStartup / ServiceShutdown
After
Watcher (pure detect)
 ├─ pollLoop() { ticker }
 ├─ check() → (Entry, isNew)
 └─ publish to channel

Persister (in HistoryService)
 ├─ consume channel
 ├─ upsert DB
 └─ emit events
Strong P1

C4: 事件名散落为字符串字面量

6 events across Go + JS

files: internal/clipboard/service.go, main.go, src/App.jsx, src/context/ClipboardContext.jsx

Problem — "clipboard-updated"、"clipboard-captured"、"window-shown"、"window-hiding"、"navigate" 以字符串形式散落在 Go 的 app.Event.Emit 和 JS 的 Events.On 中。拼错一个字不会编译错误，只会在运行时静默失败。

Solution — 在 Go 端定义事件常量（Wails v3 自动生成类型安全的 JS API）或在 JS 端集中管理事件名：

// Go: events.go
const (
    EvtClipboardUpdated  = "clipboard-updated"
    EvtClipboardCaptured = "clipboard-captured"
    EvtWindowShown       = "window-shown"
    EvtWindowHiding      = "window-hiding"
    EvtNavigate          = "navigate"
)

// JS: events.js
export const EVENTS = {
    CLIPBOARD_UPDATED: "clipboard-updated",
    WINDOW_SHOWN:      "window-shown",
    // ...
}

Benefits — Locality: 事件名集中管理，IDE 自动补全。 Leverage: 重构事件名时只需改一处。 Deletion test: 删除事件常量 → IDE 报编译/引用错误，而不是静默失败。

Worth exploring P2

C5: db.DB 作为包级全局变量

used by: main, clipboard, history, fileop (closure)

files: internal/db/db.go

Problem — internal/db 暴露 var DB *sql.DB 作为全局变量。调用方直接访问这个变量（db.DB.Exec(...)），无法 mock、无法替换，无法并发测试。

Solution — 改为 db.New(path) *DB 返回结构体，通过构造函数注入到需要它的 Service 中。当前 history、clipboard 等 Service 已经接受 *sql.DB 参数，只需把 db.DB 改为传入 dbHandle.Conn：

// Before
db.Init(path)
histSvc := history.NewService(db.DB, ...)

// After
store, _ := db.New(path)
histSvc := history.NewService(store.Conn, ...)

Benefits — Leverage: 测试时可以用 :memory: SQLite 替换。 Locality: 全局可变状态限制在 db.Store 生命周期内。

Top Recommendation

建议先做 C1（拆分 main.go）+ C2（合并适配器）+ C4（事件常量化）。 这三个改动互不依赖、可以一次完成，影响面仅限 Go 侧 2 个文件和 JS 侧 2 个文件。 收益最大：C1 让新人/下次自己读启动流程时只需 20 秒看一遍 main()，而非在 240 行里跳来跳去。 做完这三个后，main.go 应缩减到 ~60 行，外加 2 个新文件（event_names.go、app_handle.go）。

C3（拆分 ClipboardService）和 C5（去掉全局 DB）属于更深的结构性改进，建议在下一次大的功能迭代时处理。
